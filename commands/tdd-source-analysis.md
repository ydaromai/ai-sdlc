# /tdd-source-analysis — Mock App Source Code Visual Analysis

You are executing the **tdd-source-analysis** pipeline stage. This is a companion to `/tdd-mock-analysis` (Stage 3 of `/tdd-fullpipeline`). While `/tdd-mock-analysis` crawls the running mock app with Playwright to extract DOM structure and runtime behavior, this command reads the mock app **source code** to extract visual design patterns that runtime crawling cannot reliably capture: animation definitions, transition specifications, micro-interaction patterns, component variant systems, icon inventories, image assets, and page transition infrastructure.

**Why source code?** The mock app is generated by Figma AI (or similar tools). The visual design decisions — animation durations, easing curves, hover states, focus rings, component variants, icon choices — are encoded in the source code (CSS, Tailwind classes, component files). Playwright can observe the DOM at rest but cannot reliably trigger and measure every animation, hover effect, or state transition. Source code analysis provides the complete, authoritative catalog of visual design intent.

**Input:** Mock app source directory path via `$ARGUMENTS`
**Output:** `docs/tdd/<slug>/visual-system.md`

## Step 2: Animation System Analysis

Scan all source files for animation definitions. This is the highest-value extraction — animations are defined in code but nearly impossible to capture completely via runtime crawling.

### 2a. CSS Keyframe Animations

Search all `.css`, `.scss`, `.sass` files for `@keyframes` definitions. Also scan `tailwind.config.js`/`tailwind.config.ts` for `theme.extend.keyframes` (Tailwind v3) and `@theme` blocks with `--animate-*`/`--keyframes-*` tokens (Tailwind v4):

```
Patterns:
- @keyframes <name> { ... }
- theme.extend.keyframes: { <name>: { ... } }  (in tailwind.config.js/ts)
- @theme { --animate-*: ...; --keyframes-*: ...; }  (Tailwind v4)
```

For each keyframe, extract:
- Animation name
- Keyframe steps (from/to or percentage stops)
- Properties animated (transform, opacity, color, etc.)
- File path and line number

### 2b. CSS Transition Definitions

Search all `.css` and component files (`.tsx`, `.jsx`, `.vue`) for transition properties. Also scan for `@apply` directives in CSS files that compose visual utility classes (e.g., `@apply bg-primary text-white rounded-md transition-colors`) — these represent design decisions not visible in component files as inline Tailwind classes:

```
Patterns:
- transition: <property> <duration> <easing>
- transition-property / transition-duration / transition-timing-function
- Tailwind classes: transition, transition-all, transition-colors, transition-opacity,
  transition-shadow, transition-transform
- Tailwind duration: duration-75, duration-100, duration-150, duration-200, duration-300,
  duration-500, duration-700, duration-1000, duration-[<arbitrary>] (e.g., duration-[250ms])
- Tailwind easing: ease-linear, ease-in, ease-out, ease-in-out, ease-[<arbitrary>]
  (e.g., ease-[cubic-bezier(0.2,0,0,1)])
```

For each unique transition pattern, extract:
- Properties transitioned
- Duration
- Easing function
- Context (which component/element uses it)
- File path

### 2c. Animation Library Configurations

If an animation library is detected (framer-motion, motion, react-spring, GSAP):

**For framer-motion / motion:**
Search for motion component definitions and animation configs. Scan imports from both `"framer-motion"` and `"motion/react"` and `"motion"` (framer-motion v11+ uses `motion/react` as the recommended import path):
```
Patterns:
- <motion.div ... />
- <m.div ... />
- initial={{ ... }}
- animate={{ ... }}
- exit={{ ... }}
- transition={{ ... }}
- variants={{ ... }}
- AnimatePresence
- useAnimation, useMotionValue, useSpring
```

For each animation component, extract:
- Component name and file path
- Initial state (opacity, transform, etc.)
- Animate state (final values)
- Exit state (if AnimatePresence is used)
- Transition config (duration, ease, delay, stagger)
- Whether it's reusable (exported) or inline

**For react-spring:** Search for `useSpring`, `useTrail`, `useTransition`, `animated.*`.
**For GSAP:** Search for `gsap.to`, `gsap.from`, `gsap.timeline`.

### 2d. UI Library State Animations

If Radix UI, shadcn/ui, or Headless UI is detected, search for data-attribute-driven animations:

```
Patterns:
- data-[state=open]:animate-*
- data-[state=closed]:animate-*
- data-[side=top/bottom/left/right]:*
- Tailwind animate-in, animate-out, fade-in-*, fade-out-*, zoom-in-*, zoom-out-*,
  slide-in-from-*, slide-out-to-*
```

For each component with state animations, extract:
- Component name (Dialog, Sheet, Popover, Tooltip, Accordion, etc.)
- Open animation (enter)
- Close animation (exit)
- Duration and easing
- Direction variants (slide direction based on side)
- File path

### 2e. Custom Animation Components

Search for wrapper components that provide animation behavior:

```
Patterns:
- Components named *Fade*, *Slide*, *Animate*, *Transition*, *Stagger*
- Components that wrap children in motion elements
- Components using IntersectionObserver for scroll-triggered animations
```

For each custom animation component, extract:
- Component name and file path
- Props (duration, delay, direction, stagger interval)
- Default values
- Usage pattern (which pages import it)

## Step 3: Micro-Interaction Analysis

Scan for hover, focus, active, disabled, and press interaction patterns.

### 3a. Hover States

Search component files for hover-triggered style changes:

```
Patterns:
- Tailwind: hover:bg-*, hover:text-*, hover:shadow-*, hover:scale-*, hover:border-*,
  hover:opacity-*, hover:ring-*, hover:underline, hover:no-underline
- CSS: :hover { ... }
- JS: onMouseEnter/onMouseLeave with style/class changes
```

For each unique hover pattern, extract:
- Element type (button, card, link, row, icon)
- Properties changed on hover
- Specific values (e.g., `hover:bg-[#B45309]`, `hover:shadow-md`)
- Associated transition (if any)
- File path and component context

### 3b. Focus States

Search for focus-triggered style changes:

```
Patterns:
- Tailwind: focus:*, focus-visible:*, focus-within:*
- Specific: focus-visible:ring-*, focus-visible:border-*, focus-visible:outline-*
- CSS: :focus, :focus-visible, :focus-within { ... }
```

For each unique focus pattern, extract:
- Element type
- Focus indicator style (ring width, color, offset)
- Whether it uses `focus-visible` (keyboard-only) vs `focus` (all)
- File path

### 3c. Active/Press States

Search for active/press-triggered style changes:

```
Patterns:
- Tailwind: active:scale-*, active:bg-*, active:shadow-*, active:ring-*
- CSS: :active { ... }
- JS: onPointerDown/onPointerUp with transform/scale changes
```

For each unique press pattern, extract:
- Element type
- Transform/style change (e.g., `active:scale-[0.98]`)
- Duration (via associated transition)

### 3d. Disabled States

Search for disabled styling:

```
Patterns:
- Tailwind: disabled:opacity-*, disabled:cursor-*, disabled:pointer-events-*
- CSS: :disabled, [aria-disabled="true"] { ... }
- Conditional classes: isDisabled && "opacity-50"
```

### 3e. Error/Invalid States

Search for validation error styling:

```
Patterns:
- Tailwind: aria-invalid:*, invalid:*
- CSS: :invalid, [aria-invalid="true"] { ... }
- Conditional classes: error && "border-red-*", hasError && "ring-destructive"
```

For each error pattern, extract:
- Element type (input, select, textarea)
- Visual change (border color, ring color, text color)
- Error message styling (font size, color, icon)
- File path

## Step 4: Component Variant System Analysis

Scan for component variant definitions that define the full visual vocabulary.

### 4a. CVA (Class Variance Authority) Definitions

Search for `cva()` calls:

```
Pattern: cva("base-classes", { variants: { ... }, defaultVariants: { ... } })
```

For each CVA definition, extract:
- Component name
- Base classes
- Variant axes (e.g., variant, size, color)
- Each variant option with its classes
- Default variant values
- Compound variants (if any)
- File path

### 4b. Conditional Class Patterns

Search for `cn()`, `clsx()`, `classnames()`, or ternary className patterns:

```
Patterns:
- cn("base", condition && "conditional-class")
- clsx({ "class-a": isActive, "class-b": !isActive })
- className={isActive ? "active-styles" : "inactive-styles"}
```

Focus on patterns that represent visual states (active, selected, expanded, disabled) rather than layout logic. For each significant pattern, extract:
- Component name
- State variable
- Visual difference between states
- File path

### 4c. Status/State Color Mappings

Search for Record/Map/Object definitions that map status values to colors:

```
Patterns:
- Record<string, { bg: string; text: string; border?: string }>
- statusColors, stateColors, colorMap, variantColors
- switch(status) { case "active": return "bg-green-*" }
```

For each status color mapping, extract:
- Mapping name
- All status values and their colors (background, text, border)
- Usage context (which components consume this mapping)
- File path

### 4d. CSS Custom Property (Design Token) Extraction

Scan CSS files for `:root`, `@theme`, and `[data-theme]` blocks containing custom property definitions. These represent the designer's token palette and are the source of truth for Section 6.2.

```
Patterns:
- :root { --color-*: ...; --spacing-*: ...; --radius-*: ...; --shadow-*: ...; }
- @theme { --color-*: ...; }  (Tailwind v4)
- [data-theme="dark"] { --color-*: ...; }  (theme overrides)
```

For each custom property, extract:
- Property name (e.g., `--color-primary`, `--radius-lg`)
- Value (e.g., `oklch(0.205 0.01 354.116)`, `0.75rem`)
- Category (color, spacing, radius, shadow, typography — infer from prefix)
- File path
- Whether it's overridden in a dark theme block

Group by category and count unique tokens per category. This data populates Section 6.2 (Design Token Summary) in the output document.

**If no custom properties are found** (e.g., project uses only Tailwind utility classes without a theme file), set Section 6.2 to "No custom design tokens defined — project uses framework defaults" and log: `"INFO: No CSS custom properties found — Section 6.2 will note framework defaults"`.


### 7b. Character Limit

The visual system document must not exceed **50,000 characters**. If it exceeds this limit:

1. Truncate the Icon Inventory (Section 4) to top 50 icons by usage count.
2. Truncate Micro-Interaction Catalog (Section 2) to unique patterns only (no per-file duplicates).
3. If still over, summarize Component Variant System (Section 3) to variant names only (no full class listings).
4. Add a Warning at the top noting truncation.

### 7c. Write Output

1. Create the directory `docs/tdd/<slug>/` if it does not exist.
2. Write the visual system to `docs/tdd/<slug>/visual-system.md`.
3. **Do NOT commit yet** — the artifact must pass self-validation (Step 7d) before being committed.

### 7d. Self-Validation (MANDATORY)

Validate completeness against the source code and Design Brief:

**Section 1 (Animation Inventory) checks:**
1. At least 1 keyframe animation documented OR at least 1 Tailwind animation utility class found in source (`animate-spin`, `animate-pulse`, `animate-ping`, `animate-bounce`). Note: apps using only Tailwind's built-in utilities may have no project-level `@keyframes` definitions.
2. At least 1 transition pattern documented
3. If `package.json` lists an animation library, Section 1.3 must not be empty
4. If Radix UI components exist, Section 1.4 must not be empty

**Section 2 (Micro-Interactions) checks:**
1. At least 1 hover effect documented (every interactive app has hover states)
2. Focus states must document the ring/outline pattern
3. If `disabled:` classes exist in source, Section 2.4 must not be empty

**Section 3 (Component Variants) checks:**
1. If CVA is in dependencies, at least 1 CVA definition must be documented
2. If status/state color mappings exist in source, they must be documented

**Section 4 (Icons) checks:**
1. If an icon library is in dependencies, at least 5 unique icons must be documented (or at least 1 icon per route discovered, minimum 3 — whichever is higher). Simple apps with few routes may legitimately use fewer than 10 icons.
2. Status-to-icon mappings must be documented if they exist in source

**Section 5 (Loading/Transitions) checks:**
1. At least 1 loading pattern documented (skeleton or spinner)
2. If skeleton components exist in source, they must be cataloged

**FAIL condition:** If more than 2 section checks fail, log: `"SELF-VALIDATION FAILED: visual-system incomplete — failed_checks=<list>"` and re-scan the source for the missing patterns (max 2 retries).

**On PASS:** Log: `"SELF-VALIDATION PASSED: visual-system has <N> animations, <M> micro-interactions, <K> icons"`

### 7e. Commit Validated Artifact

```bash
git add docs/tdd/<slug>/visual-system.md && git commit -m "docs: add visual system for <slug>"
```

## Step 8: Critic Review (Ralph Loop)

Run a critic Ralph Loop on the visual system document.

### 8a. Critic Invocation

Spawn 3 critics in parallel (spec/orchestration file scope):

**Product Critic focus:**
- Does the visual system capture enough detail for build agents to reproduce the Figma design?
- Are animation durations and easings specific enough to implement?
- Are component variants complete (not missing obvious states)?
- Would a developer reading this document know exactly how each button, card, and form should look and behave?

**Dev Critic focus:**
- Are the Tailwind class references accurate and current?
- Are CVA patterns correctly documented?
- Are animation library APIs correctly represented?
- Are file paths valid and specific?
- Could a build agent implement these patterns from the documentation alone?

**QA Critic focus:**
- Are all interactive states documented (hover, focus, active, disabled, error)?
- Are loading states complete (every list/table should have a skeleton)?
- Are status color mappings exhaustive (every status value has colors)?
- Could a test author write visual regression tests from this document?
- Are there gaps where a micro-interaction exists in source but is not documented?

**Scoring:**
- Per-critic minimum: 8.5
- Overall minimum: 9.0
- Max iterations: 5
- Pass condition: 0 Critical + at most 2 Warnings (Warnings that refer to patterns genuinely absent from the source may be marked N/A with justification)

### 8b. Iteration Logic

Same as `/tdd-mock-analysis` Step 7b — revise the document to address findings, re-run all critics, repeat until thresholds met or max iterations reached.
